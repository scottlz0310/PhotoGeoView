## 背景と移行の理由

### Electronの課題

PhotoGeoViewは現在Electronをベースとしていますが、以下の問題に直面しています：

1. **ネイティブモジュールのビルド問題**
   - `sharp`（画像処理ライブラリ）がCI環境で頻繁にビルド失敗
   - Windows、macOS、Linuxの各プラットフォームで異なる依存関係
   - Docker環境でのクロスコンパイルが不安定

2. **electron-builderの不安定性**
   - CI/CD環境でのインストーラー生成が不安定
   - プラットフォーム固有の設定が複雑
   - 署名やコード証明書の管理が煩雑

3. **パッケージサイズの肥大化**
   - Electronバンドルが100MB以上
   - Node.jsとChromiumの両方を含む必要がある
   - ユーザーのダウンロード時間とディスク使用量が増加

4. **メモリ使用量**
   - Chromiumベースのため、メモリフットプリントが大きい
   - 画像処理と地図表示で更に増加

### Tauri移行の動機

これらの問題を解決し、より軽量で安定したアプリケーションを提供するため、Tauriへの移行を検討します。

## Tauriの利点

### 1. パフォーマンスとサイズ

- **バイナリサイズの削減**: 100MB超 → 5-10MB程度
- **起動時間の改善**: システムWebViewを使用するため高速
- **メモリ使用量の削減**: Chromiumバンドル不要

### 2. ビルドの安定性

- **Rustベースのビルドシステム**: より予測可能で安定
- **ネイティブ依存関係の管理**: Cargoエコシステムで統一
- **クロスコンパイル**: より信頼性の高いツールチェーン

### 3. セキュリティ

- **デフォルトで安全**: 最小権限の原則
- **明示的なAPI許可**: コマンドベースの通信
- **Rustの型安全性**: メモリ安全性の保証

### 4. 開発者体験

- **シンプルなAPI**: 明確で学習しやすい
- **統一されたビルドツール**: tauri-cliで一貫した体験
- **活発なコミュニティ**: 急成長中のエコシステム

## 移行における課題

### 1. 技術的課題

#### 画像処理
- **現状**: `sharp` (Node.js)を使用
- **課題**: Rustで同等の機能を実装する必要
- **候補ライブラリ**:
  - `image` crate: 基本的な画像処理
  - `kamadak-exif`: EXIF読み取り
  - `oxipng`: PNG最適化

#### ファイルシステムアクセス
- **現状**: Node.jsのfsモジュール
- **Tauriでの対応**:
  - `@tauri-apps/api/fs`: 基本的なファイル操作
  - カスタムコマンドでRust側実装

#### 地図ライブラリ
- **現状**: Leaflet + React Leaflet
- **移行**: そのまま継続利用可能（フロントエンドは変更なし）

### 2. アーキテクチャ上の課題

#### IPC通信の再設計
- **Electron**: ipcRenderer/ipcMain
- **Tauri**: invoke/listen/emit
- **影響**: すべてのバックエンド通信を書き換え

#### ストレージ
- **Electron**: electron-store
- **Tauri**: tauri-plugin-store
- **移行**: データ移行ロジックが必要

### 3. 開発環境の変更

- **Rust開発環境のセットアップ**: チーム全体で必要
- **学習曲線**: Rustの基礎知識が必要
- **ツールチェーンの変更**: Cargo、rustfmtなど

## 技術スタックの比較

| 要素 | Electron (現在) | Tauri (移行後) |
|------|-----------------|----------------|
| **バックエンド** | Node.js | Rust |
| **フロントエンド** | React + TypeScript | React + TypeScript (変更なし) |
| **ビルドツール** | electron-vite | Vite (変更なし) |
| **レンダラー** | Chromium | システムWebView |
| **パッケージング** | electron-builder | tauri-cli |
| **IPC** | ipcRenderer/ipcMain | invoke/emit |
| **ストレージ** | electron-store | tauri-plugin-store |
| **画像処理** | sharp (Node.js) | image crate (Rust) |
| **EXIF解析** | exifreader | kamadak-exif |

## アーキテクチャの変更点

### 現在のElectronアーキテクチャ

```
┌─────────────────────────────────────┐
│         Renderer Process            │
│  (React + TypeScript + Vite)        │
│                                     │
│  - UI Components                    │
│  - State Management (Zustand)       │
│  - Leaflet Map                      │
└──────────────┬──────────────────────┘
               │ IPC (ipcRenderer)
               │
┌──────────────▼──────────────────────┐
│          Main Process               │
│         (Node.js)                   │
│                                     │
│  - File System Access               │
│  - Image Processing (sharp)         │
│  - EXIF Reading (exifreader)        │
│  - Settings (electron-store)        │
└─────────────────────────────────────┘
```

### Tauri移行後のアーキテクチャ

```
┌─────────────────────────────────────┐
│         Frontend (WebView)          │
│  (React + TypeScript + Vite)        │
│                                     │
│  - UI Components                    │
│  - State Management (Zustand)       │
│  - Leaflet Map                      │
│  - Tauri API (@tauri-apps/api)      │
└──────────────┬──────────────────────┘
               │ IPC (invoke/emit)
               │
┌──────────────▼──────────────────────┐
│          Tauri Core                 │
│            (Rust)                   │
│                                     │
│  - Commands (file_operations)       │
│  - Image Processing (image crate)   │
│  - EXIF Reading (kamadak-exif)      │
│  - Settings (tauri-plugin-store)    │
│  - Window Management                │
└─────────────────────────────────────┘
```

### 主な変更点

1. **バックエンド言語**: Node.js → Rust
2. **IPC機構**: ipcRenderer/ipcMain → invoke/emit
3. **画像処理**: sharp → image crate
4. **EXIF読み取り**: exifreader → kamadak-exif
5. **設定保存**: electron-store → tauri-plugin-store
6. **レンダラー**: Chromiumバンドル → システムWebView

### フロントエンドの変更（最小限）

```typescript
// Before (Electron)
const result = await window.electron.ipcRenderer.invoke('read-image', path);

// After (Tauri)
import { invoke } from '@tauri-apps/api';
const result = await invoke('read_image', { path });
```

## 互換性とテスト戦略

### プラットフォーム互換性

| プラットフォーム | 対応WebView | 最小バージョン |
|----------------|-------------|---------------|
| Windows | Edge WebView2 | Windows 7+ |
| macOS | WKWebView | macOS 10.13+ |
| Linux | WebKitGTK | 最新ディストリビューション |

### テスト戦略

#### 1. 単体テスト
- **Rust側**: `cargo test`でRustコードのテスト

#### 2. 統合テスト
- Tauri CLIのテストフレームワークを使用
- IPC通信のエンドツーエンドテスト

#### 3. E2Eテスト
- Playwrightの継続利用
- Tauri Driverとの統合検討

#### 4. パフォーマンステスト
- メモリ使用量の測定
- 起動時間の測定
- 画像処理速度の測定

## リスクと緩和策

### リスク1: Rust学習曲線

**影響**: 開発速度の低下
**緩和策**:
- チーム向けRust研修の実施
- シンプルなタスクから段階的に実装
- Rustコミュニティのサポート活用

### リスク2: 画像処理機能の互換性

**影響**: sharpと同等の機能が実装できない可能性
**緩和策**:
- フェーズ1で徹底的な検証
- 代替ライブラリの調査
- 必要に応じてC/C++ライブラリのバインディング検討

### リスク3: プラットフォーム固有の問題

**影響**: 特定OSでの動作不良
**緩和策**:
- 早期から全プラットフォームでテスト
- CI/CDで継続的なテスト
- ベータテストプログラムの実施

### リスク4: ユーザーデータの移行

**影響**: 設定やデータの損失
**緩和策**:
- 自動移行ツールの開発
- 移行前のバックアップ推奨
- ロールバック手順の用意

### リスク5: WebViewの互換性

**影響**: 一部環境での描画問題
**緩和策**:
- Leafletの動作確認
- CSSの互換性テスト
- フォールバック戦略の準備
